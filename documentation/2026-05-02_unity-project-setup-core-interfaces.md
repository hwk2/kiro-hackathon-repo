## Implementation Choices and Reasoning

### Namespace: `RoomVisualizer`
All types use a single flat namespace. Avoids deep nesting while keeping types clearly scoped away from Unity built-ins.

### Interface-per-subsystem design
Each subsystem has a corresponding interface. This allows property-based tests to inject fakes without Unity Physics, and enables parallel development against the interface contract before implementations exist.

### `SerializableVector3` / `SerializableColor` plain structs
`UnityEngine.Vector3` and `UnityEngine.Color` are not JSON-serialisable by Newtonsoft.Json without custom converters. Plain structs with `ToVector3()`/`FromVector3()` helpers give clean round-trip serialisation with zero dependencies.

### `BlockModelData` as read-only deserialisation target
The AI Pipeline's `BlockModel` JSON uses snake_case field names. A separate `BlockModelData` class mirrors the exact schema, avoiding any transformation on the Python side and making the contract explicit.

### `AssetLibraryConfig` as ScriptableObject
Editable in the Unity Inspector without code changes, serialised as a Unity asset, and referenceable via drag-and-drop. Pre-populated with all 16 core categories from the AI Pipeline with empty asset paths.

### Assembly definition at root of `Assets/RoomVisualizer/`
Placing `RoomVisualizer.asmdef` at the folder root covers `Interfaces/`, `Models/`, and `Scripts/` subfolders in one assembly, avoiding circular references and keeping test assemblies able to reference the main assembly by name.
